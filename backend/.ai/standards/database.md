# Database Standards — ClipForge

## Current State
File-based JSON storage (`FileStorage` in `app/storage/file_storage.py`). Data persisted as `{project_id}.json` files in `_projects/` under downloads dir. Deliberate YAGNI choice for MVP.

## ORM (Future, when PostgreSQL added)
- **SQLAlchemy 2.0** — async session (`AsyncSession`).
- **Alembic** for migrations — every schema change is a new migration file.
- Run `alembic check` in CI to detect drift.

## Naming Conventions
| Element       | Convention            | Example                    |
|---------------|-----------------------|----------------------------|
| Tables        | snake_case, plural    | `projects`, `clip_renders`|
| Columns       | snake_case            | `created_at`, `youtube_url`|
| PK            | `id` (UUID)           | `id UUID PRIMARY KEY`     |
| FK            | `{table}_id`          | `project_id`               |
| Indexes       | `ix_{table}_{col}`    | `ix_jobs_status`           |
| Unique cstrs  | `uq_{table}_{cols}`   | `uq_users_email`           |
| ENUM type     | PascalCase            | `ProjectStatus`            |

## Each Table MUST Have
- `id`: UUID primary key (generated by DB: `gen_random_uuid()`).
- `created_at`: `TIMESTAMPTZ`, `server_default=func.now()`.
- `updated_at`: `TIMESTAMPTZ`, `onupdate=func.now()`.

## Indexing Strategy
- Index all FK columns (auto-join perf).
- Index columns used in WHERE / ORDER BY / GROUP BY for hot paths.
- Composite indexes for multi-column filter/sort patterns.
- Partial indexes for filtered queries: `CREATE INDEX ix_jobs_failed ON jobs(status) WHERE status = 'failed'`.
- **No over-indexing** — each index costs write perf.

## Migration Review Checklist
- [ ] Reversible? (`upgrade()` has matching `downgrade()`)
- [ ] No `DROP COLUMN` on production — 3-phase: add nullable -> backfill -> drop.
- [ ] Batch large data migrations in worker task, not migration script.
- [ ] Run `alembic autogenerate` — review generated diff carefully.
- [ ] Test migration on staging before production.

## Connection Pooling
- Engine: `create_async_engine(..., pool_size=10, max_overflow=20, pool_pre_ping=True)`.
- Pool class: `AsyncAdaptedQueuePool`.
- Worker tasks: same pool, adjust pool_size per Celery concurrency.
- Monitor `pool.wait` metric — alert if > 100ms.

## Raw SQL
- Only for complex reporting queries not expressible in ORM.
- Use SQLAlchemy `text()` with named params — NEVER string interpolation.

## Forbidden
- Storing secrets in the database.
- Raw SQL in service layer (use repository or ORM).
- Migrations without review.
- Using `sync` database operations in async context.
