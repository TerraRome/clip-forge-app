# Create SQLAlchemy Model with Migration

## Description
Create a new SQLAlchemy model class and generate the corresponding Alembic migration. Models define the database schema and ORM mapping. ClipForge currently uses `@dataclass` models (see `app/models/project.py`); this skill covers the ORM version for database-backed persistence.

## When to Use
- Adding a new database table (users, templates, clips, settings)
- Adding columns to an existing table
- Changing column types, constraints, or indexes
- Migrating from dataclass to ORM model

## Inputs
- Table name (e.g., `projects`)
- Column definitions (name, type, constraints, defaults, indexes)
- Relationships to other models
- Model class name (PascalCase singular, e.g., `ProjectModel`)

## Outputs
- ORM model at `app/models/<name>.py`
- Alembic migration version in `alembic/versions/`
- Updated `app/models/__init__.py` with imports

## Steps

1. **Create model file** at `app/models/<name>.py`. Import SQLAlchemy types: `from sqlalchemy import Column, String, Integer, Float, DateTime, Text, Enum`. Use `from app.models.base import Base` (your `DeclarativeBase` subclass). Define class inheriting from `Base`.

2. **Define columns** with explicit types and constraints. Use `String(36)` for UUIDs, `String(2048)` for URLs, `Text` for long content, `Float` for progress values. Set `nullable=False` for required fields, `default=` for defaults. Add `index=True` for frequently queried columns.

3. **Define relationships** using `relationship()` with explicit `foreign_keys`. Use `lazy="selectin"` for to-one (loaded eagerly on access), `lazy="select"` (default lazy) for to-many. Use `cascade="all, delete-orphan"` for parent-child.

4. **Add `__repr__`** for debugging: `def __repr__(self): return f"<{self.__class__.__name__}(id={self.id})>"`. Add `to_dict()` if mapping to Pydantic: `{c.name: getattr(self, c.name) for c in self.__table__.columns}`.

5. **Register in Alembic** — ensure model is imported in `app/models/__init__.py` (Alembic autogenerate scans this). Run: `alembic revision --autogenerate -m "add <table> table"`. Review the generated migration for correctness.

6. **Apply migration**: `alembic upgrade head`. Verify table exists. Test with a quick insert/query via Python shell.

## Example

```python
from sqlalchemy import Column, String, Integer, Float, DateTime, Text
from app.models.base import Base
from datetime import datetime
import uuid

class ProjectModel(Base):
    __tablename__ = "projects"

    id = Column(String(36), primary_key=True, default=lambda: str(uuid.uuid4()))
    youtube_url = Column(String(2048), nullable=False)
    num_clips = Column(Integer, nullable=False, default=3)
    subtitle_preset = Column(String(32), nullable=False, default="classic")
    status = Column(String(16), nullable=False, default="pending", index=True)
    progress = Column(Float, nullable=False, default=0.0)
    error_message = Column(Text, default="")
    clip_paths = Column(Text, default="[]")  # JSON array
    created_at = Column(DateTime, nullable=False, default=datetime.utcnow)
    updated_at = Column(DateTime, nullable=False, default=datetime.utcnow, onupdate=datetime.utcnow)
```

## Notes
- When migrating from dataclass to ORM, keep the dataclass as a domain model and create a separate ORM model with a mapping function (`.to_orm()`, `.from_orm()`) to decouple domain logic from persistence.
- Newer SQLAlchemy 2.0 syntax: `id: Mapped[str] = mapped_column(String(36), primary_key=True)`. Either style is acceptable, but be consistent within a file.
- Always review autogenerated migrations — Alembic may miss renames, detect irrelevant changes (e.g., index order), or misrepresent column types.
- Set `__tablename__` explicitly — never rely on automatic naming.
